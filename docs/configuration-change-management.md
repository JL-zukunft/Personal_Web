## 目的

本文档定义了 EasyFolio 项目环境配置变更的标准流程，确保：
- 配置变更的可追溯性
- 环境一致性
- 变更风险可控
- 问题快速回滚

---

## 适用范围

本文档适用于以下配置变更：
- 环境变量（.env 文件）
- 构建配置（vite.config.js）
- 部署配置（GitHub Actions、Vercel 配置）
- Git 忽略规则（.gitignore）

## 角色与职责

| 角色 | 职责 |
|------|------|
| **变更申请人** | 提出变更需求，填写变更申请 |
| **开发人员** | 实施配置变更，编写代码 |
| **审核人员** | 审核变更方案，评估风险 |
| **测试人员** | 执行配置验证测试 |
| **运维人员** | 执行生产环境变更部署 |

## 变更类型

### 类型 A：低风险变更

- 更新注释或文档
- 调整非关键配置项
- 添加新的可选配置项

**审批要求**：1 人审核即可

### 类型 B：中风险变更

- 修改现有配置项名称
- 调整服务器端口
- 更新依赖版本

**审批要求**：2 人审核

### 类型 C：高风险变更

- 修改 API Key 或凭证
- 变更生产环境配置
- 修改核心配置文件结构

**审批要求**：团队评审 + 负责人审批

## 变更流程

### 阶段 1：变更申请

1. 变更申请人填写《配置变更申请表》
2. 说明变更原因、内容、影响范围
3. 评估变更风险等级
4. 提交审核

### 阶段 2：变更审核

1. 审核人员评估变更方案
2. 检查是否符合配置标准
3. 确认回滚计划
4. 给出审核意见

### 阶段 3：变更实施

1. 创建功能分支：`feature/config-xxx`
2. 实施配置变更
3. 更新相关文档
4. 本地测试验证

### 阶段 4：变更测试

1. 在开发环境测试
2. 执行《配置验证清单》
3. 记录测试结果
4. 确认无问题后提交 PR

### 阶段 5：变更部署

1. PR 审核通过后合并到 main
2. 执行部署流程
3. 监控部署状态
4. 验证生产环境

### 阶段 6：变更完成

1. 更新变更记录
2. 通知相关人员
3. 归档变更文档

## 回滚计划

### 回滚触发条件

- 配置变更后出现功能异常
- 环境变量加载失败
- 部署失败
- 测试未通过

### 回滚步骤

1. 立即停止当前操作
2. 记录问题现象和错误日志
3. 执行回滚命令：
   ```bash
   # Git 回滚
   git checkout [commit-id-before-change]
   
   # 或使用 revert
   git revert [bad-commit-id]
   ```
4. 验证回滚结果
5. 分析问题原因

### 回滚验证

执行《配置验证清单》确保系统恢复正常。

## 审核与记录

### 配置变更记录

每次变更需记录以下信息：

| 字段 | 说明 |
|------|------|
| 变更ID | 唯一标识符 |
| 变更日期 | YYYY-MM-DD |
| 申请人 | 变更提出人 |
| 变更类型 | A/B/C |
| 变更内容 | 详细描述 |
| 影响范围 | 受影响的模块/环境 |
| 审核人 | 审核人员 |
| 状态 | pending/approved/rejected |
| 备注 | 其他说明 |

### 变更历史记录

所有配置变更应记录在 `docs/configuration-change-log.md` 文件中。

## 配置验证清单

### 开发环境验证

- [ ] 依赖安装成功：`npm run install:all`
- [ ] 环境变量正确加载
- [ ] 后端服务器正常启动：`npm run dev:server`
- [ ] 健康检查接口正常：`GET /api/health`
- [ ] 前端开发服务器正常启动：`npm run dev:ai`
- [ ] 前端页面正常访问
- [ ] 环境变量未泄露到前端
- [ ] .gitignore 规则生效

### 生产环境验证

- [ ] 构建成功：`npm run build:all`
- [ ] 环境变量在部署平台正确配置
- [ ] 部署流程执行成功
- [ ] 生产环境 API 正常响应
- [ ] 前端页面正常加载
- [ ] 敏感信息未暴露

### 安全验证

- [ ] .env 文件未提交到 Git
- [ ] .gitignore 包含所有敏感文件
- [ ] API Key 仅在后端使用
- [ ] 前端无硬编码密钥

## 配置标准

### 环境变量命名规范

- 使用大写字母和下划线：`COZE_API_KEY`
- 前缀标识用途：`COZE_*`、`APP_*`、`SERVER_*`
- 语义化命名，避免缩写
- 必填项无默认值，可选项提供默认值

### 配置文件格式

- 使用 UTF-8 编码
- 键值对格式：`KEY=VALUE`
- 等号前后无空格
- 注释使用 `#` 开头

### 文档要求

- 新增配置项必须更新 `.env.example`
- 变更配置必须更新相关文档
- 重大变更需在变更记录中说明
